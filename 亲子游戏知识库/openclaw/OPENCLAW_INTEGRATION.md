# OpenClaw 接入说明

这份知识库是按“本地 Markdown 文档源”来设计的，因此最适合先用 OpenClaw 直接读取目录内容，再逐步过渡到 memory 索引或向量检索。

下文中所有 `.` 路径都以当前 `./openclaw/` 目录为基准。

## 这一版为什么采用“工作区直连”

你这次明确希望用 OpenClaw 落地，而且当前只能参考现有配置文件，不能现场执行命令验证。

在这个前提下，最稳妥的方案不是猜测一堆外部索引字段，而是：

1. 直接把 OpenClaw 的默认工作区切到 `./workspace/`
2. 让 chat 直接读取这里的 Markdown 文件
3. 同时在配置里显式保留 `memory-core` / `memory-lancedb` 的插件声明

这样做的优点是：

- 接法简单，路径明确
- 不依赖未验证的 `memory root` 风格字段
- 即使你的 OpenClaw 版本未来变化，这套目录结构也仍然有效

## 本仓库已准备好的文件

- 工作区规则：`./workspace/AGENTS.md`
- 角色与灵魂：`./workspace/SOUL.md`、`./workspace/IDENTITY.md`
- 用户目标与长期记忆：`./workspace/USER.md`、`./workspace/MEMORY.md`
- 工作区本地说明：`./workspace/TOOLS.md`、`./workspace/HEARTBEAT.md`
- 知识库总览入口：`../README.md`
- 配置包说明：`./README.md`
- 独立配置包：`./openclaw.json`

## 角色文件如何把 OpenClaw 变成亲子游戏知识库

这次不是简单把一个目录挂给 OpenClaw，而是把它的“人格层”和“工作流层”也一起定制了。

作用分工如下：

- `AGENTS.md`：定义工作流、文件优先级、回答结构和写回规则
- `SOUL.md`：定义它的核心使命，不再是通用助手，而是亲子游戏策展与设计助手
- `IDENTITY.md`：固定它的角色身份和表达气质
- `USER.md`：告诉它这位使用者正在建设亲子游戏知识库，希望它围绕知识库运转
- `MEMORY.md`：沉淀长期稳定规则，例如“来源优先”“新设计先写来源笔记再出游戏卡”
- `TOOLS.md`：告诉它这套知识库目录和模板如何使用
- `HEARTBEAT.md`：留作后续知识库巡检和维护的入口

因此，OpenClaw 启动后不只是“能读这个目录”，而是会以“亲子游戏知识库设计助手”的身份来读、答、设计和写回这些文件。

## 推荐接入模式

### 模式 A：直接把知识库作为聊天工作区

这是我本次提供的主方案。

核心做法：

- 把专用配置文件放在 `./openclaw.json`
- 让这份配置把 `agents.defaults.workspace` 指向 `./workspace/`
- 让工作区根目录直接放置 `AGENTS.md`、`SOUL.md`、`IDENTITY.md`、`USER.md`、`MEMORY.md` 等角色文件
- OpenClaw chat 之后就会围绕这套知识库进行读取、检索、设计和写回

适合你的原因：

- 你要的是“通过 chat 直接回答亲子游戏内容”
- 你的内容本身就是一套结构化 Markdown 库
- 这比额外猜测 memory 挂载规则更可靠

### 模式 B：未来再升级为 memory 索引模式

如果你未来重新安装 OpenClaw，并确认你的版本支持把外部目录声明为 memory 数据源，那么可以在当前结构上继续升级。

但在这之前，没必要把接入点建立在未确认的字段名上。

## OpenClaw 回答时的内容优先级

### 用户要“可以直接玩的游戏”

优先顺序：

1. `games/` 下的 `game_card`
2. `collections/` 下的合集
3. `sources/notes/` 下的来源笔记

### 用户问“为什么这个游戏有效”或“背后的发展逻辑”

优先顺序：

1. `sources/notes/`
2. `games/`
3. `sources/books/`

### 用户要“成套推荐”

优先顺序：

1. `collections/`
2. `games/`

## 建议纳入和降权的目录

建议优先纳入：

- `games/`
- `collections/`
- `sources/notes/`

建议低优先级处理：

- `sources/books/`

建议排除：

- `_templates/`

## 为什么游戏卡应当是主要检索单元

游戏卡是最小、最可回答的知识单位。

它天然适合这类问题：

- 2 岁孩子在家可以玩什么
- 睡前安静一点的亲子游戏有什么
- 不用买材料的语言互动游戏有哪些

如果优先命中长篇来源笔记，答案往往会变得不够直接。

## 如果 OpenClaw 能读 front matter，优先使用这些字段

- `doc_type`
- `id`
- `title_zh`
- `status`
- `age_stage`
- `scenes`
- `skills`
- `materials`
- `duration_min`
- `energy_level`
- `difficulty`
- `safety_level`
- `source_refs`
- `aliases`
- `search_terms`

建议的使用顺序是：

1. 先做元数据过滤
2. 再做语义检索
3. 当用户问“能直接玩的活动”时，对 `doc_type=game_card` 进行重排加权

## 即使没有 front matter 解析，也要保持正文结构一致

每张游戏卡尽量保持同样的正文顺序：

1. 游戏简介
2. 适用场景
3. 所需材料
4. 玩法步骤
5. 引导语
6. 观察点
7. 变体玩法
8. 安全提醒
9. 来源说明

一致性会显著提高检索排序和回答整合质量。

## 推荐的内容发布策略

如果以后你把它接成对用户开放的助手，建议：

1. 默认只暴露 `reviewed` 和 `published`
2. 默认把 `draft` 排除在主检索结果之外
3. 当你需要可追溯性时，在最终答案里返回 `source_refs`

## README 中提到的配置文件包含什么

`./openclaw.json` 这份配置文件做了这些事：

- 保留你当前 `openclaw.json` 中的模型、网关和基础命令配置
- 把默认工作区改为 `./workspace/`
- 显式声明 `memory-core` / `memory-lancedb` 插件
- 显式提供插件白名单和允许路径，减少后续插件验证问题

## 为什么角色文件要放在知识库根目录

因为这次的目标不是做一个“旁路配置”，而是让 OpenClaw 直接成为这套知识库的一部分。

把角色文件放在知识库根目录的好处是：

1. 它启动后先读到的就是这套知识规则
2. 它接下来要操作的也是同一个目录里的知识文件
3. 它设计新游戏、补写来源笔记、维护合集时，不需要在不同工作区之间来回切换

## 直接替换配置后的使用方式

启用这个独立配置包后，你可以把 OpenClaw 当作一个“亲子游戏知识库聊天前端 + 游戏设计助手”来使用。

建议的首轮提问方式：

- 给我推荐 3 个适合 2 岁半、室内、10 分钟内完成的语言游戏
- 给我一个不需要材料的睡前亲子游戏
- 帮我整理一组雨天可玩的低能量亲子游戏
- 为什么这个游戏适合练习轮流等待
- 基于现有知识库模式，设计一个新的亲子游戏草案
- 把这个新游戏先写成来源笔记，再生成一张 draft 游戏卡
